## S2: v0.9.3 — Issue #4 Service-Aware Effective Family (2026-04-30)

Second-reporter bug arc on a different vector than Issue #3. Same
panel model (SDVECU) the maintainer runs, but **without perimeter
sensors provisioned** — `xSSrv` returned `[ARM, ARMNIGHT, DARM]`,
no `EST`. Every `arm_home` / `arm_away` from HA emitted
`ARMDAY1PERI1` / `ARM1PERI1` based on model-level family, panel
rejected with `code 101 / error_mpj_exception`. Disarm worked.

### Root cause

The pre-v0.9.3 design used `PANEL_FAMILIES[panel]` directly as the
perimeter gate. The classifier is per-model (SDVECU = PERI_CAPABLE
because the model *can* take perimeter sensors), but a single SDVECU
install can ship with or without sensors actually wired up. The
runtime indicator is the `EST` service in `xSSrv` — present on
maintainer's install, absent on laurafabry's. The original gate
collapsed both populations into one bucket. The `PERI` service flag
is unreliable as an alternative (absent on both panels — maintainer
fully arms perimeter without it), so a **new** discriminator was
needed: `EST` was previously dropped at parse time as "unknown",
so the data was there but invisible to the resolver.

Structurally identical to Issue #3 (SDVFAST family demotion) but
one level deeper: service granularity instead of model granularity.

### Fix

Single source of truth: `verisure_italy.models.effective_family(panel, services)`.
A `PERI_CAPABLE` install lacking `EST` demotes to `INTERIOR_ONLY`
for arm-target selection, the proto→state reverse map, and the
resolver's perimeter gate. The HA entity reads
`coordinator.active_services` at decision time so the demoted install
arms via `ARMDAY1` / `ARM1` — the commands the official Verisure IT
web app emits on the same hardware.

Five-file change set:

- `models.py` — `ServiceRequest.EST` added; `active_services()`
  retains it; `effective_family()` helper.
- `resolver.py` — `_assert_supported` uses effective family.
  `UnsupportedCommandError` distinguishes `missing={EST}` (PERI_CAPABLE
  demotion, sensors not provisioned) from `missing={PERI}` (model-
  level INTERIOR_ONLY, no perimeter hardware). Different remediation
  paths get different diagnostics.
- `coordinator.py` — pre-fetches `xSSrv` synchronously on first
  refresh, before any entity is constructed. Failure raises
  `UpdateFailed` rather than silently falling back to model-level
  family — wrong family on a no-EST install means 100% arm rejection.
- `alarm_control_panel.py` — `_panel_family` property delegates to
  `effective_family(panel, coordinator.active_services)`. All
  downstream call sites unchanged (`_ARM_TARGETS[self._panel_family]`,
  `_STATE_MAP[self._panel_family]` still keyed correctly).
- `__init__.py` exports `effective_family`.

## S3: v0.9.4 Partition-Aware Perimeter Gate (2026-05-01)

v0.9.3 didn't actually fix Issue #4 — `laurafabry` filed Issue #5
showing the same arm failure on v0.9.3, with `EST` PRESENT in services
and partition `02` empty. The EST-based `effective_family` hypothesis
was wrong: necessary but not sufficient. Investigation, full
refactor, release, and resolution in one session.

### Root cause — partition-02 enterStates is the gate

Authenticated capture of the maintainer's own SDVECU+EST web session
(`customers.verisure.it`, JWT login + SMS OTP via Chrome DevTools MCP
over SSH tunnel) produced the wire shape: the `xSSrv` envelope ships
a `configRepoUser.alarmPartitions[]` field that the live web app
already requests but `SERVICES_QUERY` did not. Web-bundle reverse-
engineering (`main.fae761fe.js`, bundle 2.4.3, function `z` at offset
390138) decoded the actual gate:

```js
case ARM_PERIMETER:    return t.enterStates > 0;  // partition '02'
case DISARM_PERIMETER: return t.leaveStates > 0;  //  ditto, leave
```

Empty `enterStates` on partition `02` (PERIMETRAL) means the user
lacks per-user perimeter permission — independent of EST. Two
SDVECU+EST installs can differ on this gate; laurafabry's profile is
the proof. EST advertises hardware-provisioning at the install level;
partitions are per-user permissions.

### Subagent-driven execution of a 13-task TDD plan

Plan written via `writing-plans` skill (`docs/plans/v0.9.4-partition-
gate.md`, 1531 lines), then executed via
`superpowers:subagent-driven-development`: fresh implementer subagent
per task, two-stage review (spec compliance + code quality) between
each. 21 commits, suite 332 → 357, pyright 0 errors throughout,
intentional-red phase between T3 and T7 documented inline. Code-
quality reviews surfaced six minor follow-ups that were applied
in-line (helper-default footgun, asymmetric test cases, dual-None
guard on `UnsupportedCommandError`, dead `_DEFAULT_SERVICES` removal,
stale Issue-#4 references in inline comments).

### What changed end-to-end

- **`models.py`**: new `AlarmPartition`, `ConfigRepoUser`, partition-
  ID constants. `effective_family(panel, alarm_partitions)` —
  signature change, no `services` parameter. Demotion now reads
  `partition[id="02"].enterStates` non-empty.
- **`graphql.py` + `responses.py`**: `SERVICES_QUERY` extended (same
  operation, additive field). `_ServiceInstallation` requires
  `configRepoUser` — fail-loud on unexpected wire drift.
- **`resolver.py`**: `CommandResolver.alarm_partitions: tuple[
  AlarmPartition, ...]` required. `_assert_supported` distinguishes
  model-level `INTERIOR_ONLY` (missing PERI service) vs runtime-
  demoted `PERI_CAPABLE` (partition-permission detail message).
- **`client.py`**: `_partitions_cache` populated alongside
  `_services_cache` from one `xSSrv` parse. New public helper
  `cached_partitions(installation)` (renamed from private when
  coordinator became a legitimate external consumer).
- **`coordinator.py`**: `alarm_partitions: tuple[AlarmPartition, ...]`
  pre-fetched on first refresh.
- **`alarm_control_panel.py`**: `_panel_family` reads
  `coordinator.alarm_partitions`. One-line argument swap.
- **`diagnostics.py`**: failure report includes a partition snapshot
  line (`alarm_partitions: 01:E[01,02]/L[01,02] 02:E[]/L[] 03:E[]/L[]`).
  Future user reports self-diagnose.
- **EST**: stays in `ServiceRequest` enum (recognised, not load-
  bearing). All capability-gate consumers removed. Pattern matches
  `PERI` (also recognised, never gated on).
