## S2: Schema-drift Hotfixes — v0.8.1 + v0.8.2 (2026-04-17)

Two user-reported config-flow crashes from the same root cause:
Verisure tightened nothing on their side and started returning
`null` for fields our Pydantic models declared as required.
Strict-by-default models did exactly what they were supposed to —
crash at the boundary — but the fields turned out to be optional.

### Issue #1 — `xSDeviceList.devices[].isActive` null (v0.8.1)

- **Fix**: `RawDevice.is_active` → `bool | None`.
- **Filter change**: camera filter in `client.py:1002` was
  `if not raw.is_active: continue` (treated null as inactive →
  silently hid devices). Changed to `if raw.is_active is False:
  continue` (only explicit False filters; null = active). Matches
  upstream
  [guerrerotook/securitas-direct-new-api](https://github.com/guerrerotook/securitas-direct-new-api/blob/f13c36e/custom_components/securitas/securitas_direct_new_api/apimanager.py#L1058)
  — three years of Spain traffic over our guess.
- **Tests added**: `test_raw_device_null_is_active` (model parse),
  `test_null_is_active_is_included` (filter behavior).

### Issue #2 — `xSInstallations.installations[].name` null (v0.8.2)

- **Fix**: nine non-load-bearing `Installation` metadata fields
  softened to `str | None`: `type`, `name`, `surname`, `address`,
  `city`, `postcode`, `province`, `email`, `phone`. Load-bearing
  fields (`number`, `panel`, `alias`) stay strict — code reads
  these for API calls and UI, parse-at-boundary still holds.
- **Tests added**: `test_null_metadata_fields_accepted`.

### Decision: hard-type only what code reads

Pre-empted all 9 metadata fields in one go rather than doing
sniper fixes per-report. Reasoning: any of those fields could
drift to null next; they are passenger data we never read; the
load-bearing trio stays strict so the parse-at-boundary contract
still holds where it matters. Issue reporter only hit `name` —
this fix unblocks the next eight too.

  → PyPI → GitHub release → comment on issue with feedback ask.
- Issue #1 comment links upstream filter at exact line + commit

## S3: Panel-Type Gate + Probe + CLI — v0.8.3 (2026-04-20)

Issue #1's reporter on a CENT panel reported arm commands "silently
failing". Root cause: `STATE_TO_COMMAND` hardcodes SDVECU-verified
mutation strings (`ARM1`, `ARM1PERI1`, `DARM1DARMPERI`, etc.) and the
client was sending them to every panel type regardless. On CENT, the
Verisure cloud accepts the mutation with `res=OK` but the panel never
receives anything — silent failure with no error surface. Fail-secure
was the tenet violated.

Full plan in `docs/plans/v0.8.3-probe-cli.md` (committed). Survived a
context compact mid-implementation.

### Strategy (rejected blind discovery)

Upstream
[guerrerotook/securitas-direct-new-api](https://github.com/guerrerotook/securitas-direct-new-api)
uses runtime command discovery: try a command, on failure retry with
variants (`ARMINTEXT1`, etc.), mark unsupported. Explicitly rejected
on fail-secure grounds — sending commands we don't understand to a
live alarm panel is the opposite of "no blind commands". Went with an
allowlist gate + read-only diagnostic probe instead.

### Library

- **`SUPPORTED_PANELS: frozenset[str] = frozenset({"SDVECU"})`** in
  `models.py` — single source of truth for verified panels.
- **`UnsupportedPanelError`** in `exceptions.py` — `VerisureError`
  subclass carrying the offending panel code.
- **`Service.attributes`** — the `attributes` wrapper was already
  being *requested* in `SERVICES_QUERY` but the model discarded it.
  Now parsed via new `ServiceAttribute` + `_ServiceAttributesWrapper`
  (mirrors the GraphQL double-nesting: `attributes { attributes { … } }`).
- **`VerisureClient.get_raw_device_list()`** — unfiltered `RawDevice`
  list from `xSDeviceList`. The pre-existing `list_camera_devices()`
  now delegates to it.
- **`VerisureClient.refresh_token`** property — public read-only
  accessor (CLI needs it to cache the session).
- **`verisure_italy/probe.py`** — new module. `run_probe(client,
  installation) -> ProbeDict`. Hits `xSSrv` + `xSDeviceList` +
  `xSStatus` only. Strictly no `xSCheckAlarm` (pings the panel, shows
  in timeline), no arm/disarm.
- **Redaction**: `_hash_numinst` (sha256 8-char prefix) + a
  `_PII_FIELDS` allowlist walked by `assert_redacted`. Devices are
  rebuilt by hand picking only non-PII fields (type, code, zoneId,
  isActive) — there's no "filter a model dict" path; the redaction is
  structural.
- **Debug logging** in `client.py`: sanitized request vars +
  response envelope at every `_execute` call, arm/disarm entry + panel
  accept + poll iterations + `_last_proto` transitions (new
  `_update_last_proto` helper logs old → new on change). Sanitization
  scrubs `password`, `idDevice`, `idDeviceIndigitall`, `uuid`.

### CLI

- **`verisure_italy/cli.py`** + `[project.scripts]` entry
  `verisure-italy-cli = "verisure_italy.cli:main"`. Subcommands:
  `login` / `probe` / `logout` — **no arm/disarm** (too dangerous as
  CLI args). Honours `VERISURE_USERNAME` + `VERISURE_PASSWORD` env
  vars for non-interactive use.
- **Session cache** at `$XDG_CACHE_HOME/verisure-italy/session.json`
  (0600). `CachedSession` is a Pydantic model — round-trips via
  `model_validate_json` / `model_dump_json`, no dict-unpack glue.
  Password never persisted; user re-enters for each `probe` or sets
  `VERISURE_PASSWORD`.
- **`scripts/verisure-italy-cli`** — bash wrapper for bare-clone use
  (`PYTHONPATH` + `python3 -m verisure_italy.cli`). pip install is
  optional.
- Both reporting paths (HA log, CLI) go through the same `run_probe`
  function — one code, one output shape. The user's explicit
  requirement: "no two paths, no two reporting codes".

### HA integration

- **`_check_panel_supported(action)`** gate in
  `alarm_control_panel.py`, called at the top of `_async_arm` and
  `async_alarm_disarm`. On unsupported panel: runs `run_probe`, logs
  the JSON between `=== VERISURE PROBE BEGIN ===` / `=== VERISURE
  PROBE END ===` markers, creates a persistent notification telling
  the user to search/open a GitHub issue (not directly #1 — any panel
  could fail), and raises `UnsupportedPanelError`. **Zero bytes sent
  to the panel.**
- Probe failure must not mask the gate — wrapped in try/except with
  `_LOGGER.exception`; gate still raises.
- Gate is action-level, not setup-level. Integration still loads on
  unsupported panels so users can see entities, read state via
  passive polling, and send us logs.

### Key finding (SDVECU live probe)

Ran the CLI against my own panel (8/8 smoke test post-deploy). From
the committed reference fixture:

- **41 services declared**, but only 4 carry `attributes`
  (`BILLS`, `ESTINV`, `SCH`, `WHATSAPP`) — none alarm-related.
- **The `ARM` service (idService=31) has `attributes: []`.** Same for
  `DARM`. Meaning: the exact mutation strings are **not** discoverable
  through `xSSrv` alone. For CENT the next step will almost certainly
  be mitmproxy on the Android Verisure app, not attribute parsing.
- Device types on SDVECU: `MG`, `VV`, `FX`, `QR`, `QP`, `VK`. No
  `CENT` entry (that's a different panel type code — lives in
  whatever the reporter's `installation.panel` returns).

Committed as both `docs/findings/panel-SDVECU-probe.json` (reference)
and `tests/fixtures/probe_sdvecu_reference.json` (regression test
parses it and asserts schema v1 + redaction). `numinst_hash` replaced
with `"REDACTED"` in both — a 7-digit numeric is brute-forceable from
the sha256 prefix.
