## S1: v0.8.4 "unavailable" flapping fix + auto-arm post-check (2026-04-20)

**Shipped:** v0.8.4 (PyPI + GitHub + HACS), plus three new automation
patterns deployed to the user's live HA config and documented in
`docs/automations.md`.

### Root cause investigation — two failure modes

Same symptom (`alarm_control_panel.verisure_alarm` → `unavailable`),
two distinct causes:

- **Mode A — brief flapping** (~8×/day): single-tick HTTP blips
  (TCP reset, 5xx) bounced the coordinator to `unavailable` for ~5s,
  recovered next poll. Annoying but not dangerous.
- **Mode B — sticky 6h outage** (2026-04-20 10:11→16:04 UTC):
  `login()` caught a GraphQL upstream Node.js error
  (`Cannot read properties of undefined (reading 'it')`) and re-raised
  it as `AuthenticationError`. HA translated that to
  `ConfigEntryAuthFailed` — Repair card, zero auto-retry. Integration
  dormant until manual HA restart. **Security-critical:** arm
  automations silently dropped for 6 hours.

### Fix — two layers

1. **HTTP-layer retry**: `_execute()` retries `APIConnectionError` and
   5xx `APIResponseError` up to 3 times inside a single coordinator
   tick. Exponential backoff 5s → 10s with ±20% jitter, cap implicit
   via tick budget. 4xx + WAF + session expiry + 2FA are **never**
   retried (each has its own recovery path). `random.random()` jitter
   not `secrets` — retry timing is not cryptographic.
2. **Remove blanket auth classification**: `login()` no longer
   converts `APIResponseError → AuthenticationError`. `AuthenticationError`
   is now reserved for confirmed credential bugs (null auth token,
   malformed JWT). Generic upstream errors propagate as
   `APIResponseError` → `UpdateFailed` → coordinator's normal
   transient-failure path. True credential problems still reach the
   Repair card.

### Why no SMS risk

Confirmed: `login()` in normal operation doesn't trigger SMS — device
trust tokens are persistent, persist across restarts. SMS only on
explicit 2FA path (separate exception type). User explicitly rejected
"automatic SMS on auth retry" design.

### Design constraints preserved

- **Fail-secure**: errors surface, not swallowed. Three retries, then
  `UpdateFailed` → `unavailable` entity state → downstream automations
  can react.
- **No binary-sensor hiding**: user chose to keep `unavailable` state
  visible. Automations opt-in to it explicitly (see post-check below).
- **No PIN**: out of scope, HA is authenticated, Verisure API requires
  credentials.

### Auto-arm failure investigation

User reported: "today the auto arm automation didn't arm and didn't
notify." Ran `verisure-investigate` skill (3 parallel agents).

Evidence:
- HA log at 14:00:47 UTC: `WARNING: Referenced entities
  alarm_control_panel.verisure_alarm are missing or not currently
  available` — service call dropped because entity was `unavailable`.
- Automation `1765795470718` `last_triggered: 14:00:47Z` — fired but
  state never changed.
- Root cause: same Mode B outage. v0.8.4 prevents the entity-state
  cause going forward.

But the investigation revealed **three design gaps independent of the
backend bug**:

1. Primary arm automation had no post-check — silent failure.
2. Safety net condition `state: disarmed` fails when state is
   `unavailable` (correctly, we do NOT want to arm a ghost entity) —
   but no notification when this happens.
3. No alert when the entity has been `unavailable` for a prolonged
   period — the user doesn't know arm/disarm is dead until they try.

### Fixes (ha-config repo)

- `1765795470718` (primary arm): `wait_for_trigger` up to 60s for
  state → `armed_away`; `choose` branches critical notify on timeout,
  normal notify otherwise.
- `1765796183149` (safety net): same post-check pattern.
- **New**: `alarm_unavailable_alert` — state trigger `unavailable
  for: 00:05:00` → critical push to both phones. 5min sweet spot:
  transient glitches (now retry-absorbed) don't fire, real outages
  do. Labeled `alarm` via `ha-ws` entity_registry update.

All three patterns documented in `docs/automations.md` with WHY
comments, not just YAML.
