## S1: v0.9.1 Post-Review Hardening Sweep (2026-04-24)

Follow-up to the v0.9.0 release. Ran the full multi-agent
`/verisure-review` against the codebase, triaged the 34 findings, and
landed fixes in three tiers — each with an on-the-live-panel E2E
cycle between deploys.

### The review

`docs/reviews/2026-04-24-codebase-review.md` — 8 parallel agents (4
line-level by scope, 4 architectural by concern) wrote independent
reports that were deduplicated into 34 findings (0 CRIT, 2 HIGH, 19
MEDIUM, 13 LOW). The review doc is the permanent record; the
release addresses the actionable subset.

### Tier A — types, schema, HA-patterns (9 commits)

Shipped first because they were the lowest regression risk.

- **H1** `dashboard.py` narrowed `except Exception` to the expected
  drift modes (`AttributeError / KeyError / TypeError / ImportError /
  OSError`) in both panel-register and setup paths. Real programming
  errors now surface.
- **H2** Config-entry schema v2 — persist the full `Installation`
  dump instead of three scalar strings with empty-string placeholders
  for missing metadata. `async_migrate_entry` synthesises v2 from v1;
  v1 entries still readable for downgrade compat. Live-tested the
  migration (entry was v1 going in, v2 coming out).
- **M1** `CommandResolver` last `@dataclass` in the codebase → Pydantic
  `BaseModel` + frozen config.
- **M2** `datetime.now()` → `datetime.now(tz=UTC)` on every
  calculation site. Overlay text keeps local time (display value,
  not comparison operand).
- **M11-revised** The probe except-Exception is DELIBERATELY broad:
  a probe crash MUST NOT mask the `UnsupportedPanelError` gate.
  Added a comment explaining the intent + regression test.
- **M12** Dashboard cards → TypedDict (`PictureEntityCard` /
  `TileCard` / `MarkdownCard` / `AlarmPanelCard` /
  `VerticalStackCard` / `ConditionalCard` / `GridSection` /
  `SectionsView` / `LovelaceConfig`). Typos in card-key names are
  now pyright errors.
- **M13** `Camera.__init__(self)` explicit call kept + documented
  — `BaseCoordinatorEntity.__init__` does NOT `super().__init__()`,
  so MRO alone does not reach `Camera.__init__` which sets
  `_cache` / `stream` / `access_tokens` required for HTTP handlers.
  The reviewer was wrong; added a comment proving so.
- **M18** `# type: ignore` comments now all carry pyright error codes
  and justifications. `test_no_blanket_type_ignore` enforces it.
- **M19** Manifest declares HA 2026.4 minimum. The thread-safety
  patterns in use (deferred `hass.async_create_task` for
  `async_call_later` callbacks) were introduced in 2026.4.

### Tier B — state machine (6 commits, highest regression risk)

Each sub-change was deployed + E2E'd before the next.

- **M3** `VerisureCoordinator.suppress_updates()` — async context
  manager, flipped by the alarm entity wrapping every mutation as
  `async with self._arm_lock, self.coordinator.suppress_updates():`.
  A 5s-interval poll landing mid-transition would previously
  overwrite the ARMING / DISARMING placeholder with a stale
  snapshot; now the coordinator short-circuits to cached data while
  suppressed. Guarded: rejects pre-first-refresh (no snapshot) +
  re-entrance.
- **M15** Typed `SameStateError(VerisureError)` — resolver raises it
  on `current == target` instead of a bare `ValueError`. Client
  arm/disarm let it pass through without a failure report; entity
  catches it at all three mutation sites, logs INFO, refreshes
  state, returns silently. With M3 already closed for the poll
  path, this is belt-and-braces for direct service-call entry points
  (REST, automations).
- **M4** Stale force-arm context auto-clears in
  `_handle_coordinator_update` when the panel has moved away from
  DISARMED via any other path (user armed via Verisure app, another
  automation, SMS, …). The stored `reference_id` / `suid` would be
  invalid once the panel moved; now it evicts itself.
- **M5** Force-context clear paths unified — the timer-expiry
  callback now routes through `_clear_force_context` rather than
  inlining the clear+notify logic a third time.
  `_expire_force_context` (silent variant) is deliberately kept:
  success paths `await async_request_refresh()` right after, and
  notifying listeners with the still-stale-because-suppressed
  `coordinator.data` would cause a one-tick render regression.
- **M6** Fail-secure on `OperationTimeoutError` — arm/disarm timeout
  no longer reverts the UI to a guessed prior state. Entity now
  sets `_attr_alarm_state = None` (UNKNOWN in HA's
  alarm_control_panel contract) and falls through to
  `coordinator.async_request_refresh()`. The real panel state
  resolves on the next poll. `OperationFailedError` (explicit
  panel rejection) keeps the revert behaviour — unambiguous.
- **M17** Structured audit log for every force-arm attempt. One
  grep-stable `FORCE_ARM_AUDIT {…}` JSON line per attempt (success,
  noop, timeout, failure) with reference_id, suid, mode, bypassed
  zones, proto-before, duration, and on non-success error-class +
  message. Satisfies the CLAUDE.md audit-trail requirement for the
  most security-sensitive mutation in the codebase.

### Tier C — error paths (1 bundled commit)

- **M9** Session-expired re-login now catches network errors in the
  inner retry block (`APIConnectionError` / `APIResponseError` /
  `WAFBlockedError`) and raises `UpdateFailed`. Previously they
  bubbled out unhandled → DataUpdateCoordinator logged "Unexpected
  error fetching" + gave up. Also: avoids the wrong signal of
  `ConfigEntryAuthFailed` (which triggers HA's re-auth dialog) for
  what is actually a network blip.
- **M10** Config flow maps `VerisureError` → `cannot_connect` and
  `ValidationError` → `unknown` instead of letting them surface as
  raw tracebacks on login / 2FA / reauth / reconfigure screens.
  New translation strings added to `strings.json` + `en.json`.
- **M8** Coordinator camera discovery now catches `ValidationError`
  (upstream listing schema drift) and degrades to zero cameras with
  a log line, instead of taking down the alarm poll.
- **M16** Thumbnail timestamp parse fallback logs a WARNING before
  using local time. Was silent; drift would have gone unnoticed.

### Decisions

- **Tier'd deploy cadence, not one big bang** — regression risk
  scales with blast radius. A timeout-path fix (Tier B) that revealed
  a bug after Tier A shipped would have been clean to bisect; a
  bug-on-bug across 19 findings would not have been.
- **Live E2E between tiers, not mocked** — the user explicitly
  authorised disarm → arm_home → disarm → arm_away → disarm →
  capture cycles on the real installation. Every tier got a fresh
  kill-switch-off / on cycle and a 6/6 camera capture, which is a
  real-world test the unit suite can't replicate.
- **`SameStateError` not "catch ValueError"** — the review suggested
  the latter; doing it would have masked two legitimate resolver
  raises (unknown panel, cross-perimeter armed transition) that
  SHOULD surface loud. Typed exception, one specific catch.
- **Reviewer was wrong about M13** — audit-trail for cases where
  dogma clashes with reality. `BaseCoordinatorEntity.__init__` does
  not super-init, so MRO alone leaves `Camera.__init__` unreached;
  dropping the explicit call would have broken the camera HTTP
  handlers. Documented + kept.

## S2: v0.9.2 INTERIOR_ONLY HA State Mapping (Issue #3, 2026-04-24)

First-reporter bug from `alan210874` against the fresh-out-of-the-door
v0.9.1 build. On `SDVFAST` (INTERIOR_ONLY family, no perimeter
sensors) every arm button surfaced `UnsupportedCommandError: Panel
'SDVFAST' cannot honour 'ARMDAY1PERI1': missing active service(s):
PERI`. Fail-secure was working; UX was broken.

### Root cause

`custom_components/verisure_italy/alarm_control_panel.py` hard-coded
`_PARTIAL_PERIMETER` (`PARTIAL/ON`) and `_TOTAL_PERIMETER` (`TOTAL/ON`)
as the arm targets for `arm_home` / `arm_away`. On a `PERI_CAPABLE`
panel that's correct; on an `INTERIOR_ONLY` panel it forced the
resolver to pick a `*_PERIMETER` command that the family gate then
correctly rejected. Reverse map had the same shape-bug: proto `P`/`T`
were labelled `ARMED_CUSTOM_BYPASS` — true on peri-capable (someone
armed only the interior axis), wrong on interior-only (that IS
home/away).

### The fix (approach A)

Entity-layer only — no changes to resolver, client, or models.

- `_ARM_TARGETS: dict[PanelFamily, dict[AlarmControlPanelState, AlarmState]]`
  — arm_home/arm_away target states keyed by panel family.
  PERI_CAPABLE keeps `PARTIAL/ON` + `TOTAL/ON`; INTERIOR_ONLY collapses
  to `PARTIAL/OFF` + `TOTAL/OFF`.
- `_STATE_MAP: dict[PanelFamily, dict[AlarmState, AlarmControlPanelState]]`
  — reverse map split the same way. PERI_CAPABLE retains
  `CUSTOM_BYPASS` for the three mixed states (E/P/T). INTERIOR_ONLY
  only knows the perimeter-OFF subset; perimeter-involving AlarmStates
  are absent by design — a panel returning one raises `KeyError` and
  crashes loud (fail-secure on family-impossible state).
- `VerisureAlarmPanel._panel_family` property reads
  `PANEL_FAMILIES[coordinator.installation.panel]`. Methods consult
  it at call time, not constructor time — one lookup cost per
  transition, keeps the module top-level data static.

### Web-bundle validation

Before writing code, grep of `/tmp/verisure-web-bundle/main.2.4.2.js`
confirmed the approach mirrors the Verisure IT web app:

- `R(e)` classifier: SDVFAST/SDVFSW → `false` (INTERIOR_ONLY); every
  other panel → `true`. Matches `PANEL_FAMILIES` 1:1.
- Arm resolver `w({alarmMode, currentMode, isCU})`:
  `ARM_DAY → "ARMDAY1"`, `ARM_TOTAL → "ARM1"`. The `PERI`-suffixed
  commands (`ARMDAY1PERI1`, `ARM1PERI1`) are only ever emitted for
  `ARM_PARTIAL_ARM_PERIMETER` / `ARM_TOTAL_ARM_PERIMETER` — alarm
  modes the UI never offers on `isCU === false` panels.
- `availableModes` for INTERIOR_ONLY:
  `{DISARM, ARM_DAY, ARM_NIGHT, ARM_TOTAL}`. No perimeter variants.
