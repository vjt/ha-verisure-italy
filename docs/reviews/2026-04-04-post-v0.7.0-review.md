## HIGH Findings

### H1. Config flow proceeds to empty 2FA when device already validated
**File:** `config_flow.py:97-101`
**Flagged by:** 1/8 (integration-core)

When `TwoFactorRequiredError` is caught, `validate_device(None, None)` is
called. If device is already authorized, it returns `(None, [])`. Code
unconditionally proceeds to `async_step_2fa_phone()` — user sees empty
phone list, gets stuck. Same pattern in reauth and reconfigure flows.

**Fix:** When `otp_hash is None`, skip to `async_step_installation()` (or
`_update_entry` for reauth/reconfigure).

### H2. No guard against concurrent arm/disarm/force-arm operations
**File:** `alarm_control_panel.py:157-277`
**Flagged by:** 1/8 (state-management)

`_arm_in_progress` is a boolean, not a mutex. Two concurrent operations
(e.g., arm_away + disarm from different triggers) can both run, sending
conflicting commands to the physical panel. First operation's `finally`
clears `_arm_in_progress` while second is still awaiting, allowing a poll
to sneak through and write stale state.

**Fix:** Replace boolean with `asyncio.Lock`. Reject or queue if locked.

### H3. ValidationError uncaught in coordinator polling path
**File:** `coordinator.py:199-248`
**Flagged by:** 1/8 (boundaries)

`_async_update_data` catches all `VerisureError` subtypes but not
`pydantic.ValidationError`. If Verisure changes the xSStatus response
shape, an inscrutable Pydantic traceback replaces meaningful error
handling. The alarm entity goes unavailable with no explanation.
Also: `TwoFactorRequiredError` during re-auth login doesn't trigger
`ConfigEntryAuthFailed`.

**Fix:** Catch `ValidationError` → `UpdateFailed`. Catch
`TwoFactorRequiredError` → `ConfigEntryAuthFailed`.

### H4. Own `ClientSession` instead of HA's shared session
**File:** `coordinator.py:143`, `config_flow.py:64`
**Flagged by:** 1/8 (HA-patterns)

Both coordinator and config flow create standalone `aiohttp.ClientSession()`.
Bypasses HA's SSL context, proxy settings, lifecycle management. Config flow
session leaks if user abandons mid-2FA.

**Fix:** Use `async_get_clientsession(hass)` or `async_create_clientsession(hass)`.

### H5. Dashboard manipulates Lovelace internals
**File:** `dashboard.py`
**Flagged by:** 1/8 (HA-patterns)

Imports `LovelaceStorage`, pokes into `hass.data["lovelace"].dashboards`.
Any HA Lovelace refactor breaks this at import time. Also missing
`"lovelace"` in `manifest.json` dependencies.

**Fix:** Accept as known risk with try/except guard, or serve static
YAML-mode dashboard. Add `"lovelace"` to manifest dependencies.

### H6. No HA-level disarm PIN requirement
**File:** `alarm_control_panel.py:93`
**Flagged by:** 1/8 (security)

`_attr_code_arm_required = False` and no `code_disarm_required`. Any HA
user (including guests on shared tablets) can disarm the alarm without
a code. `force_arm` service is also unrestricted.

**Fix:** Add `_attr_code_disarm_required = True` with
`_attr_code_format = CodeFormat.NUMBER`.

### H7. `_check_disarm_status_once` discards error field
**File:** `client.py:923-931`
**Flagged by:** 1/8 (client)

Converts `DisarmResult` to `OperationResult` which has no `error` field.
Disarm poll errors lose their `code` and `type`. Human notification says
"Panel rejected operation" with no diagnostic info.

**Fix:** Extract `error.code`/`error.type` before conversion, or raise
`OperationFailedError` directly with the real error codes.

## MEDIUM Findings

### M1. Naive `datetime.now()` in 7 production locations
**Files:** `alarm_control_panel.py:137,304`, `coordinator.py:346,435,438`, `client.py:337,435`
**Flagged by:** 5/8 agents

v0.7.0 fixed JWT/auth datetimes but missed force context (`created_at`),
camera timestamps, request ID generation, and login timestamp. The force
context comparison (`now() - ctx.created_at`) is fragile across DST.

### M2. Force context expiry coupled to poll boundaries
**File:** `alarm_control_panel.py:132-143`
**Flagged by:** 1/8 (state-management)

120s TTL checked only at poll intervals. At 300s poll interval, context
could live 7+ minutes. Stale `reference_id`/`suid` may be server-expired.
Fix: `async_call_later` for deterministic expiry.

### M3. Alarm stuck in ARMING during force-arm decision window
**File:** `alarm_control_panel.py:160-178`
**Flagged by:** 1/8 (state-management)

After `ArmingExceptionError`, `_attr_alarm_state` stays ARMING for the
entire decision window. Panel is DISARMED. Fix: revert to last known
state before setting force context.

### M4. `camera_entities: list[object]` with `hasattr` duck typing
**File:** `coordinator.py:184,196-197`
**Flagged by:** 5/8 agents

Should be a `CameraRefreshable` Protocol like `ForceArmable`.

### M5. Services schema mismatch — services.yaml lies
**File:** `__init__.py:120-138`, `services.yaml`
**Flagged by:** 1/8 (HA-patterns)

`services.yaml` declares `entity_id` selector. Code uses `vol.Schema({})`
which rejects it. Service only works when called bare.

### M6. Missing `strings.json` entries for reauth flow
**File:** `config_flow.py`, `strings.json`
**Flagged by:** 1/8 (HA-patterns)

Reauth steps (`reauth_confirm`, `reauth_2fa_phone`, `reauth_2fa_code`)
have no translation strings. User sees blank labels.

### M7. Config flow missing exception coverage for non-login calls
**File:** `config_flow.py` (send_otp, list_installations, get_services)
**Flagged by:** 1/8 (boundaries)

Network failure during 2FA/installation steps shows generic "unknown error".

### M8. `TwoFactorRequiredError` unhandled in coordinator auth refresh
**File:** `coordinator.py:208`
**Flagged by:** 1/8 (boundaries)

If device auth is revoked server-side, login during poll raises
`TwoFactorRequiredError` instead of triggering reauth flow.

### M9. `_get_exceptions` returns empty list on timeout
**File:** `client.py:830-844`
**Flagged by:** 2/8 agents

User sees "blocked by open zones:" with no zone names.

### M10. `_last_proto` empty on first arm after startup
**File:** `client.py:169`
**Flagged by:** 1/8 (security)

First arm/disarm sends `currentStatus=""` to API. Should populate from
`get_general_status` during first refresh.

### M11. `UnexpectedStateError` not surfaced as persistent notification
**File:** `coordinator.py:220-222`
**Flagged by:** 1/8 (security)

Unknown proto code logs error but doesn't create persistent notification
like arm/disarm failures do.

### M12. Fire-and-forget tasks without names or background tracking
**Files:** `__init__.py:48`, `camera.py:46`
**Flagged by:** 2/8 agents

Should use `config_entry.async_create_background_task` for proper
lifecycle tracking and cancellation on unload.

## Trajectory Review

**What we built:** Complete Verisure Italy replacement in HA — alarm
control, force-arm, cameras, auto-managed dashboard, typed client library.
v0.7.0 resolved all CRITICAL and HIGH items from the first review.

**Does it serve the mission?** Yes. Core alarm loop is solid. Type safety
significantly improved. Exception coverage comprehensive in alarm paths.

**What improved since v0.6.0:**
- ForceArmContext/ForceArmable eliminate the untyped dict/Any holes
- _STATE_MAP crash-loud with startup assertion
- Full exception coverage in arm/disarm/force-arm
- Reauth flow for expired credentials
- Camera entities properly use CoordinatorEntity

**What remains:**
- Concurrency gap (H2) — most actionable security fix
- Config flow 2FA logic bug (H1) — blocks users
- ValidationError in polling (H3) — API change resilience
- Disarm PIN (H6) — defense in depth for multi-user
- ClientSession management (H4) — HA best practice
- Datetime consistency (M1) — 7 naive datetime.now() calls

**Direction:** Fix H1 (logic bug, one-liner), H2 (asyncio.Lock), H3
(catch ValidationError+TwoFactorRequiredError in coordinator). H6 is a
product decision (PIN requirement). H4/H5 are HA best practices that
reduce breakage risk but aren't security bugs.
