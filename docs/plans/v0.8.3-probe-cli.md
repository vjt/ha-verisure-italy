# Plan: v0.8.3 — Probe CLI + Panel-Type Discovery

Status: design approved, ready for execution after context compact.

## Problem

Issue #1 reporter on a **CENT** panel (not SDVECU). Config flow works
post-v0.8.1. **Arm commands silently fail** — panel never receives the
command. Verisure app confirms no command reached the panel.

Our code currently has no concept of panel type; `STATE_TO_COMMAND`
hardcodes SDVECU-verified mutation strings (`ARM1`, `ARMDAY1PERI1`,
`DARM1DARMPERI`, etc.). Upstream (guerrerotook/securitas-direct-new-api,
cloned at `~/code/ha/securitas-direct-new-api`) uses runtime command
discovery with fallback — **explicitly rejected** by user per the "no
guessing, no blind commands" security tenet. Fail-secure mandates
explicit per-panel command maps, built from observed data.

## Strategy

1. **Don't send commands to unknown panels.** Ever. Gate `_async_arm`
   and `async_alarm_disarm` on a `SUPPORTED_PANELS` allowlist. Unknown
   panel = `UnsupportedPanelError`, HA notification, zero bytes sent.
2. **Build a probe** — read-only library function that dumps everything
   we can learn about a panel via GraphQL (installations, services with
   attributes, device list, alarm state). Shared code path between HA
   (emitted to log on unsupported-panel detection) and a CLI tool
   (`verisure-italy-cli`).
3. **CLI is optional.** Same data is available in HA logs. CLI exists
   for fast iteration: user pulls repo, logs in, runs probe, sends
   output — no HA restart needed.
4. **Codify local SDVECU reference** — run probe against my own
   installation, commit redacted output as `docs/findings/panel-SDVECU-probe.json`
   and unit-test fixture. Baseline for diffing against CENT later.

## Scope — In

- Probe library function (`verisure_italy/probe.py`)
- CLI (`verisure_italy/cli.py`) with `login` / `probe` / `logout`
- Entry point `verisure-italy-cli` via `pyproject.toml [project.scripts]`
- Shell wrapper `scripts/verisure-italy-cli` for bare-clone usage
- `Service.attributes` Pydantic parsing (currently requested in GraphQL
  but discarded)
- `VerisureClient.get_raw_device_list()` — unfiltered devices (existing
  method filters to cameras)
- `SUPPORTED_PANELS: frozenset[str] = {"SDVECU"}` in models
- `UnsupportedPanelError` exception
- HA gate in `custom_components/verisure_italy/alarm_control_panel.py`:
  `_async_arm` + `async_alarm_disarm` raise + emit probe to log on
  unknown panel
- Debug logging in `client.py` arm/disarm paths: request vars (sanitized),
  response envelope, each poll iteration, `_last_proto` transitions
- Unit tests: probe fixture parsing, redaction, CLI arg parsing
- `docs/findings/panel-SDVECU-probe.json` (redacted reference)
- `docs/findings/panel-types.md` — how to request support for a new panel
- CHANGELOG entry

## Scope — Out

- Arm/disarm in CLI (too dangerous — one wrong flag = unintended disarm)
- Automatic panel-type detection beyond the allowlist gate (no fallback
  discovery)
- Reading HA storage to extract credentials (re-login via CLI is fine;
  tests the flow)
- CENT command mapping (separate PR after probe data arrives)

## Redaction (strictly enforced in probe.py)

**Dropped entirely:** `numinst` (replaced by `sha256(numinst)[:8]` as
`numinst_hash`), `phone`, `email`, `name`, `surname`, `address`, `city`,
`postcode`, `province`, `alias`, device `name`, `serialNumber`,
`auth_token`, `capabilities` JWT, `referenceId`.

**Kept (non-PII, essential for diagnosis):** `panel`, installation `type`,
all Service fields (especially `attributes` — the prize), device `type`,
`code`, `zoneId`, `subtype`, `idService`, `isActive`, alarm proto codes +
timestamps.

Redaction is a function in `probe.py`, covered by a unit test that
asserts every sensitive field is scrubbed on a fixture input.

## CLI design

```
verisure-italy-cli <command> [args]

Commands:
  login              Authenticate interactively. Prompts email, password,
                     handles OTP. Persists session to ~/.cache/verisure-italy/session.json
  probe              Load session, run probe, write redacted JSON to stdout.
                     Args: --installation N (default: first), --pretty
  logout             Delete cached session
  --help, --version
```

Session cache format (`~/.cache/verisure-italy/session.json`):
```json
{
  "username": "...",
  "refresh_token": "...",
  "device_id": "...",
  "uuid": "...",
  "id_device_indigitall": "...",
  "saved_at": "ISO8601"
}
```

Chmod 0600. Warn if world-readable. Path is `XDG_CACHE_HOME` aware.

## SUPPORTED_PANELS gate (HA layer)

In `custom_components/verisure_italy/alarm_control_panel.py`:

```python
from verisure_italy import SUPPORTED_PANELS, UnsupportedPanelError, run_probe

async def _async_arm(self, target, mode):
    if self._installation.panel not in SUPPORTED_PANELS:
        # Emit probe to HA log with boundary markers so user can copy
        probe = await run_probe(self._client, self._installation)
        _LOGGER.warning(
            "=== VERISURE PROBE BEGIN ===\n%s\n=== VERISURE PROBE END ===",
            json.dumps(probe, indent=2),
        )
        raise UnsupportedPanelError(
            f"Panel type '{self._installation.panel}' not yet supported. "
            f"Search github.com/vjt/ha-verisure-italy/issues for your "
            f"panel type, or open a new issue with the probe output from "
            f"your HA log (search for 'VERISURE PROBE BEGIN')."
        )
    # existing logic
```

`UnsupportedPanelError` subclasses `HomeAssistantError` so HA surfaces it
in the UI. Probe runs on the HA event loop — already async, no executor
needed.

**No arm attempt is made.** Panel receives zero bytes.

## Debug logging (client.py)

New debug logs at these points (all at `logging.DEBUG`, no PII):

1. `_execute` — log operation name + request variables (minus auth fields)
   + response body truncated to 4KB.
2. `arm()` entry — log `target_state`, resolved `command`, `currentStatus`,
   `panel`.
3. `arm()` response — log `res`, `msg`, `referenceId`.
4. `_check_arm_status_once` — log counter, `res`, `status`, proto.
5. `disarm()` mirror the same three.
6. `_last_proto` setter — log `old → new`.

User enables via HA `logger: custom_components.verisure_italy: debug` and
`verisure_italy: debug`.

## Risks & mitigations

| Risk | Mitigation |
|------|-----------|
| Probe exposes PII | Unit test asserts redaction on every sensitive field |
| Probe makes side-effectful call | Only read-only queries: xSInstallations, xSSrv, xSStatus, xSDeviceList. NO xSCheckAlarm (pings panel, timeline entry). NO xSArmPanel/xSDisarmPanel. |
| SDVECU users break on upgrade | SUPPORTED_PANELS includes "SDVECU" — existing users unchanged |
| CLI stores creds insecurely | 0600 session file, refresh token only (not password), XDG_CACHE_HOME |
| Schema drift breaks tests | `schema_version` field, test loads fixture, parse on drift = loud fail |
| Probe too large for HA log | Current estimate ~5-20KB JSON, fits. If grows: fallback to `/config/verisure_probe_<ts>.json`. Not now. |

  mapping is committed.
